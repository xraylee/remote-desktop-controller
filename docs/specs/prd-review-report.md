# RDCS PRD v1.0 + 原型稿 完整审核报告

**审核日期**：2026-06-25
**审核范围**：PRD v1.0、原型稿 v1/v2/v3/flow、市场分析、脑暴 V1/V2、AGENTS.md
**审核结论**：PRD 整体结构完整、交互逻辑清晰，存在 **6 项需修复的问题** 和 **8 项建议优化项**

## 一、PRD 内部一致性审核

### ✅ 通过项

- 功能清单与交互流程基本对齐，P0/P1/P2 优先级划分合理
- 三层连接架构（L1/L2/L3）在全文中保持一致描述
- 设备码（9 位纯数字）与邀请码（4 位数字）的区分表在全文一致
- 非功能需求与验收标准对应（如延迟 <10ms 在 AC-03 中有验证）

### ❌ 需修复项

| # | 问题 | 位置 | 严重度 |
|---|------|------|--------|
| **F1** | 免费版限速矛盾：风险章节（§13）提到"免费用户限速 720P"，但 AGENTS.md 和脑暴 V1 明确要求"免费版不限速、1080P/60FPS"。PRD 自身也未在非功能需求中定义免费版画质标准 | §13 vs AGENTS.md | 🔴 高 |
| **F2** | Web 控制台"权限配置"（P1）与"客户端无角色区分"存在潜在矛盾——如果客户端无角色，权限配置的作用范围是什么？需要在 PRD 中明确"权限配置仅影响 Web 控制台操作权限"还是"也影响客户端行为" | §3.3 vs §5.2 | 🟡 中 |
| **F3** | 数据埋点中 `connection_initiated` 的"扫码"连接方式在 MVP 中已降级为 P1，但埋点事件仍包含"扫码"选项 | §10 | 🟢 低 |

### ⚠️ 建议优化

| # | 建议 | 位置 |
|---|------|------|
| **O1** | 补充免费版 vs 付费版的功能差异表（当前散落在风险章节和脑暴文档中，PRD 无统一定义） | §2 或 §3 新增 |

## 四、用户体验合理性审核

### ✅ 优秀设计

- **双卡片分区布局**：将"让别人连我"和"我连别人"两种意图空间分离，降低认知负担
- **邀请码命名**：比"临时口令"更直觉，符合中国用户"快递取件码"的心智模型
- **设备码 vs 邀请码区分表**：PRD §4.2 的对比表非常清晰，避免开发混淆
- **性能指标动态反馈**：颜色编码（绿/黄/红）+ 横幅提示，用户无需理解技术细节即可感知网络状况
- **连接状态弱化**：L1/L2/L3 作为底部小标签，不抢占主界面焦点

### ⚠️ 建议优化

| # | 问题 | 建议 |
|---|------|------|
| **U1** | 邀请码"刷新"按钮的功能不够明确——用户为什么要刷新？ | 补充刷新场景说明（如"怀疑泄露时可主动刷新"） |
| **U2** | "附近设备"列表的排序逻辑未定义——按信号强度？按名称？按最近连接？ | 补充排序规则（建议：在线优先 + 按最近连接时间排序） |
| **U3** | Web 控制台"团队创建"流程缺少引导——首个用户如何知道要去 Web 控制台创建团队？ | 补充客户端内的引导入口（如首次启动时提示"管理你的团队 →"链接） |
| **U4** | 远程会话工具栏按钮过多（8-9 个），小屏幕下可能溢出 | 建议定义按钮优先级和溢出策略（如低频操作收入"更多"菜单） |

## 五、技术可行性与风险审核

### ✅ 合理约束

- WebRTC 作为画面传输强制要求——方向正确，生态成熟
- 核心引擎跨平台复用 + 平台适配层分离——架构合理
- MVP 聚焦 macOS 单平台——降低初始复杂度
- 三层连接架构（LAN/P2P/Relay）——行业标准方案

### ⚠️ 需关注的风险

| # | 风险 | 当前缓解 | 补充建议 |
|---|------|---------|---------|
| **R1** | 中转成本是最高概率风险（PRD 标注"高"），但缺少具体的成本测算模型 | "免费用户限速 720P" | 建议补充单用户月均带宽成本估算 + 盈亏平衡点 |
| **R2** | WebRTC 在弱网环境下的表现可能不如预期（丢包、抖动） | 未提及 | 建议补充 WebRTC SVC/自适应码率策略的需求描述 |
| **R3** | macOS 屏幕录制权限是系统级弹窗，用户拒绝后无法再次触发（需手动到系统设置） | "引导到系统设置" | 建议补充：客户端检测权限状态，未授权时在界面显著位置提示 |
| **R4** | 4 位邀请码仅 10000 组合，5 次错误锁定 30 分钟——但如果攻击者分布式尝试（不同 IP），锁定机制可能无效 | "5 次错误锁定" | 建议补充：设备码级别的速率限制（如单设备码每小时最多接受 N 次连接请求） |
| **R5** | PRD 未提及"自部署"选项，但 AGENTS.md 明确"100% Self-Hosted by Default" | 无 | 需要决策：是否保留自部署能力？如果是，建议列为 V2.0+ 规划 |

## 八、总结

**PRD 质量评分**：**7.5/10**

**优势**：
- 结构完整，覆盖了 PRD 应有的所有章节
- 交互流程详尽，异常处理路径完善
- 双卡片布局、邀请码命名、性能动态反馈等 UX 决策经过多轮迭代验证
- 技术选型约束合理，不过度干涉技术设计

**待改进**：
- 免费版策略需要明确统一（当前 PRD 内部 + 与 AGENTS.md 存在矛盾）
- 安全要求覆盖不全（缺少 TOTP、TLS 1.3、加密存储）
- 自部署 vs SaaS 的定位需要团队层面决策
- 验收标准和原型覆盖可进一步补充
