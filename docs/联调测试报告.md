## 2026-07-02 补充：信令协议对齐验证（snake_case 修复）

**测试环境：** 服务端 + 主控客户端同机部署于 192.168.31.50，目标被控端 192.168.31.194

### 根因与修复

之前端到端连接失败的根因是**客户端与服务端的信令消息 `type` 判别字段大小写不一致**：

- 服务端 (Rust serde)：`#[serde(tag = "type", rename = "connect_request")]` → **snake_case**
- 客户端 (Dart freezed)：默认 union key 为 **camelCase**（`connectRequest`、`useInvite`…）

服务端收到 camelCase 的 `type` 会当作未知消息，导致信令无法路由。

**修复：** 在 `signaling_message.dart` 为 `@Freezed` 增加 `unionValueCase: FreezedUnionCase.snake`，
使客户端序列化/反序列化的 `type` 全部为 snake_case，与服务端 serde 完全对齐。

配套改动：
- `signaling_service.dart`：`connect()` / `requestConnection()` 增加连接状态守卫，未连接时抛错（供上层重试）。
- `session_providers.dart`：连接前对 signaling 重连 + 发送 `connect_request` 增加最多 3 次重试。
- `home_page.dart`：「连接远程设备」改为弹出邀请码输入框，走 `use_invite` 流程。

### ✅ 验证结果（针对真实运行的服务端）

服务状态（Docker，全部 healthy）：signaling `ws://192.168.31.50:8443/ws`、API `:8080`、relay UDP `:3478`。
> 注意：8443 当前为**明文 WS**（`requireTls:false`），客户端配置应使用 `ws://` 而非 `wss://`。

关键证据（客户端日志 + 服务端日志双向印证）：

这证明 snake_case 判别字段在**双向**都正确工作，且服务端能在两个真实客户端连接之间正确路由信令。

## 修复的问题

### 1. 黑屏问题
**原因：** `build_rust_lib.sh` 中 `BUILT_PRODUCTS_DIR` 未设置时，库文件路径错误
```
❌ Debug/Contents/Frameworks/librdcs_core.dylib
✅ Debug/rdcs_client.app/Contents/Frameworks/librdcs_core.dylib
```

**修复：** 完善路径判断逻辑

### 2. 服务器地址配置错误
**原因：** 默认配置使用了错误的端口
```
❌ ws://localhost:8080
✅ wss://localhost:8443
```

**修复：** 更新 `app_config.dart` 默认值
